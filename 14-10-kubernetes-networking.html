Kubernetes Network Security

ğŸ“‹ What You'll Learn

Kubernetes networking is fundamentally different from traditional networking. By default, all pods can communicate with all other pods - making network policies essential for security.

  • Kubernetes networking model
  • Network Policies for segmentation
  • CNI plugins and security features
  • Ingress security
  • Egress controls
ğŸ§  Memory Aid: Kubernetes Network Security
ğŸ˜ï¸ The Neighborhood Analogy

Kubernetes networking is like a gated community:

  • Cluster = The entire gated community
  • Namespace = Different neighborhoods within
  • Pod = Individual houses (with their own IP address)
  • Service = Community directory - "Contact the Smith family" resolves to a house
  • NetworkPolicy = Neighborhood rules - "No visitors after 10 PM"
  • Ingress = Main gate guard - controls who enters from outside
ğŸ”’ Critical K8s Security Concept

By default, Kubernetes is WIDE OPEN:

  • Every pod can talk to every other pod (no fences between houses!)
  • You MUST add NetworkPolicies to restrict traffic
  • CNI plugin must support NetworkPolicies (Calico, Cilium - NOT Flannel alone)
ğŸ¯ K8s Security Checklist:
1. Default deny NetworkPolicy in every namespace
2. Explicitly allow only required traffic
3. Use a CNI that supports policies (Calico/Cilium)
4. Don't forget to allow DNS (kube-system, port 53)!

  name: secure-ingress
  annotations:
    # TLS redirect
    nginx.ingress.kubernetes.io/ssl-redirect: "true"
    # Rate limiting
    nginx.ingress.kubernetes.io/limit-rps: "10"
    nginx.ingress.kubernetes.io/limit-connections: "5"
    # WAF via ModSecurity
    nginx.ingress.kubernetes.io/enable-modsecurity: "true"
    nginx.ingress.kubernetes.io/modsecurity-snippet: |
      SecRuleEngine On

    # Security headers
    nginx.ingress.kubernetes.io/configuration-snippet: |
      add_header X-Frame-Options "SAMEORIGIN";
      add_header X-Content-Type-Options "nosniff";
      add_header Strict-Transport-Security "max-age=31536000";
spec:
  ingressClassName: nginx
  tls:
  - hosts:
    - api.example.com
    secretName: api-tls-cert
  rules:
  - host: api.example.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: api-service

              number: 80

Kubernetes Network Security Checklist

Control Description
Default deny Apply deny-all NetworkPolicy to all namespaces
Namespace isolation Restrict cross-namespace traffic
Egress controls Limit outbound to required destinations
Use CNI with policies Calico or Cilium, not just Flannel
Ingress TLS Terminate TLS at ingress, encrypt in-cluster
Service mesh Consider Istio/Linkerd for mTLS
Private API server Don't expose API server to internet
Audit logging Enable API server audit logs
ğŸš¨ Common Kubernetes Network Mistakes
  • No NetworkPolicies: Default allows all pod-to-pod traffic
  • Flannel without policies: Flannel doesn't support NetworkPolicy
  • Forgot DNS egress: Deny-all blocks DNS, breaking apps
  • Public API server: K8s API exposed to internet
  • No egress controls: Compromised pods can reach anywhere
ğŸ’¡ Interview Tip

Question: "How do you secure networking in Kubernetes?"

Answer: "Kubernetes networking is open by default - all pods can reach all other pods. I'd implement defense in depth starting with NetworkPolicies. First, apply a default-deny policy to every namespace, then explicitly allow only required communication paths. This requires a CNI that supports policies - Calico or Cilium, not Flannel. For egress, I'd restrict outbound traffic to only required external services and always allow DNS to kube-system. At the ingress layer, use an ingress controller with TLS termination, rate limiting, and WAF capabilities. For sensitive workloads, add a service mesh like Istio for mTLS between pods. Keep the API server private - don't expose it to the internet. Finally, use namespace isolation to separate environments and limit blast radius of any compromise."
