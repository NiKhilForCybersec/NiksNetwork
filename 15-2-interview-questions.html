Network Security Interview Questions

This comprehensive collection of interview questions covers all major network security topics. Questions range from fundamental concepts to advanced troubleshooting scenarios, suitable for L1 SOC Analyst through Senior Security Engineer roles.

  • 100+ questions with detailed answers
  • Organized by topic area
  • Real-world scenario questions
  • Tips for structuring your answers
  • Common follow-up questions

Networking Fundamentals

Q1: Explain the OSI model and where different security devices operate.

Answer: The OSI model has 7 layers. Security devices operate at different layers:

  • Layer 7 (Application): WAF, Proxy servers, Application firewalls - inspect HTTP/HTTPS content
  • Layer 4 (Transport): Stateful firewalls, IDS/IPS - track TCP/UDP connections
  • Layer 3 (Network): Router ACLs, packet filter firewalls, VPN gateways - filter by IP
  • Layer 2 (Data Link): Switches with port security, 802.1X NAC - MAC-based controls

Key point: NGFWs operate across layers 3-7, providing deep packet inspection.

Q2: What's the difference between TCP and UDP? When would you use each?

Answer:

  • TCP: Connection-oriented, reliable delivery, ordered packets, error checking. Uses 3-way handshake (SYN, SYN-ACK, ACK). Used for HTTP, HTTPS, SSH, FTP - where data integrity matters.
  • UDP: Connectionless, no delivery guarantee, faster, lower overhead. Used for DNS, NTP, VoIP, streaming - where speed matters more than guaranteed delivery.

Security implication: UDP is easier to spoof (no handshake verification), making it popular for DDoS amplification attacks (DNS, NTP). TCP SYN floods target the handshake mechanism.

Q3: Explain the TCP three-way handshake and how SYN flood attacks exploit it.

Answer:

Three-way handshake:

  1. Client sends SYN (synchronize) packet
  2. Server responds with SYN-ACK
  3. Client sends ACK - connection established

SYN Flood Attack: Attacker sends thousands of SYN packets with spoofed source IPs. Server allocates resources for each half-open connection waiting for the ACK that never comes. This exhausts server's connection table.

Mitigations: SYN cookies (don't allocate resources until handshake completes), rate limiting, firewall rules, dedicated DDoS protection.

Q4: What is ARP and how can it be exploited?

Answer:

ARP (Address Resolution Protocol): Maps IP addresses to MAC addresses on a local network. When a device needs to communicate, it broadcasts "Who has IP X?" and the owner responds with its MAC address.

ARP Spoofing/Poisoning: Attacker sends fake ARP replies, associating their MAC with another IP (like the gateway). Traffic intended for the gateway goes to the attacker instead - enabling man-in-the-middle attacks.

Mitigations:

  • Dynamic ARP Inspection (DAI) on switches
  • Static ARP entries for critical systems
  • DHCP Snooping (validates ARP against DHCP bindings)
  • Network segmentation (VLANs)
Q5: Explain VLAN hopping and how to prevent it.

Answer:

VLAN Hopping: Technique to access traffic on other VLANs that should be isolated. Two main methods:

  1. Switch Spoofing: Attacker's device negotiates a trunk link with the switch using DTP, gaining access to all VLANs
  2. Double Tagging: Attacker sends frame with two 802.1Q tags. First tag matches native VLAN (stripped by first switch), second tag directs to target VLAN

Prevention:

  • Disable DTP on all access ports: switchport nonegotiate
  • Set native VLAN to unused VLAN (not VLAN 1)
  • Explicitly configure access ports: switchport mode access
  • Tag native VLAN on trunks: vlan dot1q tag native

Firewall Questions

Q6: What is the difference between a stateless and stateful firewall?

Answer:

Stateless (Packet Filter):

  • Examines each packet independently
  • Decides based only on source/dest IP and port
  • No memory of previous packets
  • Fast but limited security
  • Requires explicit rules for return traffic

Stateful:

  • Tracks connection state (NEW, ESTABLISHED, RELATED)
  • Maintains state table of active connections
  • Automatically allows return traffic for established connections
  • Can detect out-of-state packets (potential attacks)
  • More secure but requires more resources

Example: With stateful, you create one rule "allow outbound HTTPS" and return traffic is automatically permitted. Stateless would need rules in both directions.

Q7: What makes a Next-Generation Firewall (NGFW) different from traditional firewalls?

Answer: NGFWs have three key capabilities beyond traditional firewalls:

  1. Application Awareness: Identifies applications regardless of port (detects Skype on port 443, not just "HTTPS"). Uses deep packet inspection and behavioral analysis.
  2. User Identity Integration: Maps traffic to users via AD/LDAP integration. Enables policies like "Allow Marketing group to use social media."
  3. Integrated Threat Prevention: Built-in IPS, antimalware, URL filtering, sandboxing - not separate devices.

Additional NGFW features: SSL/TLS decryption, threat intelligence feeds, cloud-based analysis, integration with SIEM/SOAR.

Q8: How do you design firewall rules following best practices?

Answer:

  1. Default deny: Block everything by default, explicitly allow required traffic
  2. Least privilege: Allow only the minimum necessary access
  3. Specific before general: More specific rules first, catch-all rules last
  4. Document everything: Each rule should have description, owner, business justification
  5. Regular review: Audit rules quarterly, remove unused rules
  6. Use groups: Group similar sources/destinations for easier management
  7. Avoid "any any": Be specific about sources, destinations, and services
  8. Enable logging: Log denies and critical allows for visibility
Q9: A user reports they can't access a web application. How do you troubleshoot if it's a firewall issue?

Answer: Systematic approach:

  1. Verify the symptom: What exactly fails? Connection timeout? Refused? Error page?
  2. Check connectivity basics: Can you ping the server? Traceroute shows path?
  3. Check firewall logs: Search for traffic from user's IP to app's IP/port. Look for deny entries.
  4. Policy lookup: Use firewall's policy test feature (Palo Alto: "test security-policy-match")
  5. Packet capture: Capture on firewall to see if traffic arrives and what happens
  6. Check NAT: If NAT involved, verify correct translation
  7. Session table: See if session is being created/dropped

Commands (Palo Alto):

  • show session all filter source 10.1.1.100 destination 192.168.1.50
  • test security-policy-match source 10.1.1.100 destination 192.168.1.50 protocol 6 destination-port 443
Q10: Explain SSL/TLS inspection on firewalls. What are the security and privacy considerations?

Answer:

How it works:

  1. Firewall terminates the client's SSL connection (acts as server)
  2. Decrypts and inspects the traffic
  3. Creates new SSL connection to actual server (acts as client)
  4. This is essentially a controlled man-in-the-middle

Requirements:

  • CA certificate deployed to all endpoints (they trust firewall-signed certs)
  • Significant CPU resources on firewall
  • Certificate handling for sites with pinned certs

Considerations:

  • Security benefit: Can detect malware, C2, data exfiltration in HTTPS
  • Privacy: Must exclude sensitive categories (healthcare, banking, HR systems)
  • Legal: May need employee notification depending on jurisdiction
  • Performance: Reduces throughput significantly (often 80%+ reduction)

VPN Questions

Q11: What is the difference between IKE Phase 1 and Phase 2?

Answer:

Phase 1 (ISAKMP SA):

  • Establishes secure, authenticated channel between VPN peers
  • Negotiates encryption, hash, DH group, authentication method
  • Creates management tunnel to protect Phase 2 negotiation
  • Uses Main Mode (6 messages, secure) or Aggressive Mode (3 messages, faster but less secure)
  • Longer lifetime (8 hours typical)

Phase 2 (IPSec SA):

  • Negotiates parameters for actual data encryption
  • Defines "interesting traffic" (proxy IDs/selectors)
  • Creates the tunnel that encrypts user traffic
  • Multiple Phase 2 tunnels can use one Phase 1
  • Shorter lifetime (1 hour typical)
  • Optional PFS (new DH exchange for each rekey)
Q12: A VPN tunnel is not coming up. Walk me through your troubleshooting process.

Answer: Systematic approach:

  1. Verify basic connectivity:
    • Can peers reach each other? (ping remote public IP)
    • Is UDP 500 and 4500 open? (check intervening firewalls)
  2. Check Phase 1:
    • Do proposals match exactly? (encryption, hash, DH group, lifetime)
    • Is authentication correct? (PSK matches, certificates valid)
    • Check for "no proposal chosen" errors
  3. Check Phase 2:
    • Do proxy IDs match? (common cause - mismatch in network definitions)
    • Do transform sets match?
    • Is PFS configured the same on both sides?
  4. Common issues:
    • NAT-T issues if behind NAT (need UDP 4500)
    • Asymmetric routing
    • MTU/fragmentation problems

Debug commands: show vpn ike-sa, show vpn ipsec-sa, enable IKE debugging

Q13: What is Perfect Forward Secrecy (PFS) and why is it important?

Answer:

Definition: PFS ensures that if a long-term key (like Phase 1 key) is compromised, attackers cannot decrypt previously captured traffic.

How it works: With PFS enabled, each Phase 2 rekey performs a new Diffie-Hellman exchange, generating fresh keys independent of Phase 1 keys.

Without PFS: Phase 2 keys are derived from Phase 1 keys. If Phase 1 is compromised, attacker can calculate all Phase 2 keys and decrypt historical captures.

With PFS: Each Phase 2 has its own DH exchange. Compromising one session doesn't affect others.

Best practice: Always enable PFS using DH Group 14 or higher. The small performance overhead is worth the significant security benefit.

WAF & Web Security Questions

Q14: What is the difference between a WAF and a network firewall?

Answer:

Network Firewall:

  • Operates at Layers 3-4 (or 3-7 for NGFW)
  • Controls traffic based on IP, port, protocol
  • Protects network perimeter
  • Stops unauthorized access

WAF (Web Application Firewall):

  • Operates at Layer 7 (Application layer)
  • Understands HTTP/HTTPS protocol deeply
  • Inspects request content, headers, parameters, body
  • Protects against application attacks: SQL injection, XSS, CSRF
  • Typically deployed in front of web servers

Key difference: Network firewall might allow HTTPS (port 443), but can't see the SQL injection in the request body. WAF can inspect the decrypted content and block the attack.

Q15: Explain SQL injection and how a WAF detects it.

Answer:

SQL Injection: Attacker inserts malicious SQL code into input fields that get executed by the database.

Example: Login form with username field:

  • Normal: john_doe
  • Attack: ' OR '1'='1' --
  • Query becomes: SELECT * FROM users WHERE username='' OR '1'='1' --'
  • This returns all users, bypassing authentication

WAF Detection Methods:

  1. Signature-based: Looks for patterns like ' OR, UNION SELECT, ; DROP TABLE
  2. Anomaly-based: Detects unusual characters in parameters (single quotes, semicolons where not expected)
  3. Positive security model: Only allows known-good input patterns

WAF action: Block request, log attempt, optionally return custom error page

Q16: How do you handle false positives in a WAF?

Answer:

  1. Identify the false positive:
    • Review blocked request details in WAF logs
    • Determine which rule triggered
    • Understand why legitimate traffic matched
  2. Options for remediation:
    • Create exception: Whitelist specific URL/parameter combination
    • Adjust rule: Modify detection sensitivity if configurable
    • Disable rule: Last resort - document why and compensating controls
  3. Best practices:
    • Start in detection/monitor mode before blocking
    • Test with real traffic to tune before production
    • Create narrow exceptions (specific URL + parameter, not broad)
    • Document all exceptions with business justification
    • Review exceptions periodically

IDS/IPS Questions

Q17: What is the difference between IDS and IPS?

Answer:

IDS (Intrusion Detection System):

  • Passive monitoring - copies of traffic (span port or TAP)
  • Detects and alerts on suspicious activity
  • Does NOT block traffic
  • No impact if fails (traffic continues to flow)
  • Used for visibility and forensics

IPS (Intrusion Prevention System):

  • Inline deployment - traffic passes through it
  • Detects AND blocks malicious traffic
  • Can drop packets, reset connections, block IPs
  • Higher risk - misconfiguration can cause outages
  • Requires careful tuning to avoid false positives

Modern reality: Most deployments use IPS (inline), but start in IDS/detection mode while tuning, then enable blocking once confidence is high.

Q18: Explain signature-based vs anomaly-based detection.

Answer:

Signature-based:

  • Matches traffic against known attack patterns (signatures)
  • Very accurate for known threats (low false positives)
  • Cannot detect zero-day/novel attacks
  • Requires regular signature updates
  • Example: Looking for /etc/passwd in web requests

Anomaly-based:

  • Learns normal behavior baseline
  • Alerts on deviations from normal
  • Can detect unknown/zero-day attacks
  • Higher false positive rate
  • Requires training period
  • Example: User suddenly downloading 10GB when normal is 100MB

Best practice: Use both in combination. Signatures for known threats, anomaly detection for unusual behavior.

Architecture & Design Questions

Q19: Explain the difference between North-South and East-West traffic security.

Answer:

North-South Traffic:

  • Traffic between internal network and external (Internet)
  • Traditional perimeter security focus
  • Protected by edge firewalls, proxies, WAFs
  • Represents minority of total traffic (~20%)

East-West Traffic:

  • Traffic between internal systems/segments
  • Server-to-server, VM-to-VM communication
  • Represents majority of data center traffic (~80%)
  • Often unmonitored/uncontrolled in traditional networks
  • Critical for preventing lateral movement

Modern approach: Zero Trust assumes breach; implement micro-segmentation and internal firewalls to control East-West traffic. If an attacker compromises one server, they shouldn't freely access others.

Q20: Design a DMZ architecture for a company hosting public web applications.

Answer: Classic three-tier DMZ with dual firewalls:

Architecture:

  1. External Firewall: Between Internet and DMZ
    • Allows: HTTPS (443) to WAF/Load Balancer
    • Denies: All other inbound
  2. DMZ Zone:
    • WAF (inspects HTTP traffic)
    • Load Balancer
    • Web Servers (front-end only)
    • Reverse Proxy
  3. Internal Firewall: Between DMZ and Internal
    • Allows: Web servers ‚Üí App servers on specific ports (8080)
    • Allows: App servers ‚Üí Database on port 1433/3306
    • Denies: Direct DMZ ‚Üí Database
  4. Internal Zones:
    • App Server tier
    • Database tier (most protected)

Key principles: Defense in depth, each tier only talks to adjacent tier, database never directly exposed to DMZ.

Q21: What is Zero Trust Architecture and how would you implement it?

Answer:

Core principle: "Never trust, always verify." Assumes breach has already occurred; verify every access request regardless of location.

Key concepts:

  • No implicit trust: Being on corporate network doesn't grant access
  • Verify explicitly: Authenticate and authorize every request
  • Least privilege: Minimum access needed for the task
  • Assume breach: Segment access to limit blast radius

Implementation pillars:

  1. Identity: Strong authentication (MFA), identity-based policies
  2. Device: Device health verification, MDM compliance
  3. Network: Micro-segmentation, software-defined perimeter
  4. Application: Application-level authentication, API security
  5. Data: Classification, encryption, DLP

Technologies: ZTNA solutions (Zscaler ZPA, Prisma Access), identity providers, micro-segmentation (VMware NSX), CASB

Scenario-Based Questions

Q22: You notice unusual outbound traffic from a server to an external IP on port 443. Walk me through your investigation.

Answer: This could be legitimate or C2 traffic. Investigation steps:

  1. Gather information:
    • What server is this? What's its normal function?
    • What's the destination IP? (WHOIS, threat intel lookup)
    • How much data? How long has it been happening?
  2. Check threat intelligence:
    • Is the IP known malicious? (VirusTotal, threat feeds)
    • Is it a known cloud service? (AWS, Azure ranges)
  3. Analyze the traffic:
    • If SSL decryption enabled: What's the actual content?
    • Certificate details: Who issued it? What domain?
    • Traffic pattern: Beaconing? Regular intervals = suspicious
  4. Check the server:
    • What process is making the connection?
    • Any new/unusual processes?
    • Check for persistence mechanisms
  5. Contain if malicious:
    • Block the IP at firewall
    • Isolate the server
    • Initiate incident response
Q23: Your company is moving to the cloud. How would you design network security for AWS workloads?

Answer: Defense-in-depth approach for AWS:

  1. VPC Design:
    • Separate VPCs for different environments (prod/dev/staging)
    • Public subnets for load balancers, private for applications
    • Dedicated subnet for databases (no Internet access)
  2. Network ACLs (Stateless):
    • Subnet-level filtering
    • Broad allow/deny (backup layer)
  3. Security Groups (Stateful):
    • Instance-level micro-segmentation
    • Reference other SGs (web SG ‚Üí app SG ‚Üí db SG)
    • Principle of least privilege
  4. AWS Network Firewall / Third-party NGFW:
    • For IPS, advanced threat prevention
    • Centralized egress filtering
  5. AWS WAF:
    • Protect public-facing applications
    • OWASP managed rules
  6. Connectivity:
    • VPN or Direct Connect to on-prem
    • Transit Gateway for multi-VPC
    • PrivateLink for AWS services
