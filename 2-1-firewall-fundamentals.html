Firewall Fundamentals

ğŸ“‹ What You'll Learn

Firewalls are the cornerstone of network security, acting as the first line of defense between trusted internal networks and untrusted external networks. This guide covers everything from basic concepts to enterprise deployment strategies.

  • What is a firewall and how it evolved over generations
  • Packet filtering vs stateful inspection vs application-aware
  • Firewall placement in network architecture
  • North-South vs East-West traffic protection
  • Hardware, software, and virtual firewall options
  • Throughput considerations and sizing

What is a Firewall?

A firewall is a network security device that monitors and controls incoming and outgoing network traffic based on predetermined security rules. It establishes a barrier between a trusted internal network and untrusted external networks, such as the Internet.

1st Generation: Packet Filtering Firewalls

The first firewalls examined packets in isolation, making decisions based solely on source/destination IP addresses and port numbers. They were fast but couldn't understand the context of a connection.

Characteristic Description
OSI Layer Layer 3 (Network) and Layer 4 (Transport)
Decision Criteria Source/Destination IP, Source/Destination Port, Protocol
State Awareness None - each packet evaluated independently
Performance Very fast - minimal processing overhead
Limitations Cannot detect application-layer attacks, vulnerable to IP spoofing
Modern Example Router ACLs, iptables without connection tracking
Linux iptables - Packet Filter Example 
# Simple packet filtering rules (stateless)
# Allow incoming HTTP traffic
iptables -A INPUT -p tcp --dport 80 -j ACCEPT

# Allow incoming HTTPS traffic
iptables -A INPUT -p tcp --dport 443 -j ACCEPT

# Block all other incoming traffic
iptables -A INPUT -j DROP

# View current rules
iptables -L -n -v

Stateful firewalls track the state of network connections (TCP streams, UDP communication) and make decisions based on the context of the traffic, not just individual packets.

âœ… Best Practice

Always use stateful inspection for perimeter firewalls. Modern firewalls are stateful by default, but understanding the difference is crucial for:

  • Troubleshooting connection issues (check state tables)
  • Understanding why return traffic is allowed
  • Sizing firewalls (state table capacity is a limiting factor)
  • Detecting asymmetric routing issues

3rd Generation: Application Layer Firewalls

Application layer firewalls (also called proxy firewalls) can understand and inspect the actual application protocols like HTTP, FTP, and SMTP. They can detect and block malicious content within allowed protocols.

4th & 5th Generation: NGFW (Next-Generation Firewalls)

NGFWs combine traditional firewall capabilities with advanced features like application awareness, user identity integration, SSL/TLS inspection, and integrated threat prevention.

Feature Traditional Firewall NGFW
Application Identification Port-based only (port 80 = HTTP) Deep inspection (identifies Zoom, Slack, etc. regardless of port)
User Identity IP-based policies only Integrates with AD/LDAP for user-based policies
SSL Inspection Cannot see encrypted traffic Decrypts, inspects, re-encrypts (MITM)
IPS Separate device Integrated
Threat Intelligence Manual updates Real-time cloud-based updates
Sandboxing Not available Cloud or on-prem sandbox for unknown files

Common Question: "What makes a Next-Generation Firewall different from a traditional firewall?"

Answer: NGFWs have three key differentiators:

  1. Application Awareness (App-ID): Identifies applications regardless of port, protocol, or encryption. A traditional firewall sees "port 443" while an NGFW sees "Dropbox file sharing."
  2. User Identity (User-ID): Maps IP addresses to usernames via AD/LDAP integration, enabling policies like "Allow Sales team to access Salesforce."
  3. Content Inspection (Content-ID): Inspects actual payload for threats, including decrypting SSL traffic, scanning for malware, and applying URL filtering.

Bonus point: Mention specific vendor terminology like Palo Alto's "App-ID, User-ID, Content-ID" or Fortinet's "Security Fabric."

North-South vs East-West Traffic

Understanding traffic flow patterns is essential for effective firewall placement and policy design.

Aspect North-South Traffic East-West Traffic
Direction Between internal network and external (Internet) Between internal segments/servers
Volume ~20% of total traffic (decreasing) ~80% of total traffic (increasing)
Security Focus Block external threats, prevent data exfiltration Prevent lateral movement, contain breaches
Firewall Type Perimeter NGFW Internal/Segmentation FW, Micro-segmentation
Example Policies "Block all inbound except HTTPS to web servers" "Only web tier can talk to app tier on port 8080"
âš ï¸ Security Warning

Don't neglect East-West security!

Most organizations focus heavily on perimeter (North-South) security but leave internal traffic unchecked. This allows attackers who breach the perimeter to move laterally with ease.

Key insight: The majority of modern attacks involve lateral movement. APTs and ransomware typically compromise one system, then spread internally. Without East-West controls, a single compromised workstation can lead to full domain compromise.

Firewall Throughput and Sizing

Selecting the right firewall size requires understanding various performance metrics and how they relate to your traffic patterns.

Key Performance Metrics

Metric Description Considerations
Firewall Throughput Maximum bandwidth with basic L3/L4 filtering Lab conditions, rarely achieved in production
Threat Prevention Throughput Bandwidth with IPS, AV, URL filtering enabled More realistic, 30-60% of firewall throughput
SSL Inspection Throughput Bandwidth when decrypting/inspecting HTTPS Most intensive; 10-30% of base throughput
Concurrent Sessions Maximum simultaneous connections tracked Critical for large networks; state table exhaustion
New Sessions/Second Rate of new connection establishment Important for bursty traffic, web servers
IPSec VPN Throughput Bandwidth for encrypted VPN traffic Relevant for site-to-site and remote access
ğŸ”´ Critical Sizing Consideration

Vendor datasheets show IDEAL conditions. Real-world performance is typically 40-60% of published numbers. When sizing:

  1. Calculate your required throughput with ALL features enabled
  2. Add 30-50% headroom for growth
  3. Account for SSL inspection (currently 80-90% of web traffic is HTTPS)
  4. Consider peak vs average traffic

Rule of thumb: If you need 2 Gbps of threat prevention throughput, look at firewalls rated for 4-5 Gbps.

ğŸ“… Day-to-Day Tasks

As a Security Engineer managing firewalls, your typical tasks include:

  • Rule Management: Creating, modifying, and reviewing firewall rules
  • Log Analysis: Reviewing traffic logs for anomalies and incidents
  • Policy Optimization: Identifying unused rules, consolidating redundant rules
  • Threat Monitoring: Reviewing IPS alerts and threat logs
  • Health Monitoring: Checking CPU, memory, session counts
  • Change Management: Documenting and approving rule changes
  • Compliance Audits: Generating reports for PCI-DSS, HIPAA, etc.
  • Troubleshooting: Investigating connectivity issues
