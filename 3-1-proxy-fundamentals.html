Explicit vs Transparent Proxy

Aspect Explicit Proxy Transparent Proxy
Client Config Required (browser/system settings) Not required
User Awareness Users know about proxy Users may not know
Deployment PAC file, GPO, manual config Network redirect (WCCP, PBR)
HTTPS Handling CONNECT method Requires inline interception
Authentication Easy (proxy auth headers) Complex (IP-based or captive portal)
Bypass Risk Users can remove config Harder to bypass (network-level)

PAC Files and WPAD

PAC File Example 
// PAC (Proxy Auto-Configuration) File
// Browsers download this and execute to determine proxy for each URL

function FindProxyForURL(url, host) {
    
    // Direct connection for internal sites
    if (isPlainHostName(host) ||
        shExpMatch(host, "*.internal.company.com") ||
        isInNet(dnsResolve(host), "10.0.0.0", "255.0.0.0") ||
        isInNet(dnsResolve(host), "172.16.0.0", "255.240.0.0") ||
        isInNet(dnsResolve(host), "192.168.0.0", "255.255.0.0")) {
        return "DIRECT";
    }
    
    // Direct connection for localhost
    if (host == "localhost" || host == "127.0.0.1") {
        return "DIRECT";
    }
    
    // Direct connection for specific domains (e.g., video conferencing)
    if (shExpMatch(host, "*.zoom.us") ||
        shExpMatch(host, "*.teams.microsoft.com") ||
        shExpMatch(host, "*.webex.com")) {
        return "DIRECT";
    }
    
    // Use primary proxy, failover to secondary
    return "PROXY proxy1.company.com:8080; PROXY proxy2.company.com:8080; DIRECT";
}
Method Description Pros/Cons
Manual Config User manually enters proxy in browser Simple but doesn't scale
PAC File URL Browser downloads PAC from specific URL Flexible, conditional logic
WPAD Auto-discover PAC via DNS/DHCP Zero-touch but security concerns
GPO Push settings via Active Directory Centralized, managed, Windows only

Authentication Methods

Method Description Use Case
IP-based Trust traffic from specific IP ranges Transparent proxy, simple environments
Basic Auth Username/password (Base64 encoded) Simple but credentials sent in clear (use HTTPS)
NTLM Windows integrated authentication Domain-joined Windows clients
Kerberos Ticket-based (more secure than NTLM) Domain environments, SSO
SAML/OAuth Federation with identity provider Cloud proxies, modern auth

ğŸ’¡ Interview Tip

Common Question: "Explain SSL inspection and its challenges."

Answer:

  • What it does: Proxy decrypts HTTPS traffic, inspects content, re-encrypts to client using proxy's certificate
  • Why needed: 80%+ of traffic is encrypted - without inspection, threats hide in HTTPS
  • Challenges:
    • Certificate deployment (CA must be trusted by all clients)
    • Certificate pinning (some apps reject proxy certs, need bypass)
    • Privacy concerns (inspecting user traffic)
    • Performance (decryption/re-encryption CPU intensive)
    • Legal/compliance (some traffic shouldn't be inspected - banking, healthcare)
