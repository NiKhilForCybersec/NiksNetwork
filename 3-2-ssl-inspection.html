Bypass Categories

âš ï¸ Categories to Bypass SSL Inspection

Some traffic should NOT be decrypted for legal, privacy, or technical reasons:

  • Financial/Banking: Legal and compliance requirements
  • Healthcare: HIPAA considerations
  • Government: Sensitive communications
  • Certificate pinning: Apps that pin certificates will break
  • VPN clients: Enterprise VPN connections
  • Video conferencing: Performance and privacy
  • Software updates: Windows Update, antivirus updates

OpenSSL - Certificate Verification 
# Check what certificate is presented (with SSL inspection)
openssl s_client -connect www.google.com:443 -servername www.google.com

# Look for Issuer - should show your internal CA if inspection is active
# Issuer: CN = Your-Company-SSL-Inspection-CA

# Without inspection, you'd see:
# Issuer: C = US, O = Google Trust Services, CN = GTS CA 1C3

# Check certificate chain
openssl s_client -connect www.google.com:443 -showcerts

# Verify specific certificate
openssl x509 -in cert.pem -text -noout

ğŸ’¡ Interview Tip

Common Question: "How does SSL inspection work and what are the challenges?"

Answer framework:

  1. How it works: "The inspection device acts as a man-in-the-middle. It terminates the client's SSL session, inspects the traffic, then creates a new SSL session to the server. It presents clients with a certificate signed by an internal CA."
  2. Certificate deployment: "The internal CA must be trusted by all clients - typically deployed via GPO for Windows, MDM for mobile devices."
  3. Challenges: "Certificate pinning breaks some apps, there are privacy/legal concerns for certain categories (banking, healthcare), and it adds CPU overhead."
  4. Best practice: "Create bypass rules for sensitive categories and certificate-pinned applications, and monitor device performance."
