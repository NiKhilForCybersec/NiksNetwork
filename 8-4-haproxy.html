frontend https_front
    bind *:443 ssl crt /etc/ssl/certs/example.com.pem
    mode http
    
    # Security headers
    http-response set-header Strict-Transport-Security "max-age=31536000"
    http-response set-header X-Frame-Options "SAMEORIGIN"
    http-response set-header X-Content-Type-Options "nosniff"
    
    # ACLs for routing
    acl is_api path_beg /api
    acl is_admin path_beg /admin
    acl is_websocket hdr(Upgrade) -i WebSocket
    
    # Rate limiting (requires stick-table)
    stick-table type ip size 100k expire 30s store http_req_rate(10s)
    http-request track-sc0 src
    http-request deny deny_status 429 if { sc_http_req_rate(0) gt 100 }
    
    # Routing rules
    use_backend api_servers if is_api
    use_backend admin_servers if is_admin
    default_backend web_servers

Load Balancing Algorithms

Algorithm Description Use Case
roundrobin Sequential distribution Stateless, equal servers
leastconn Fewest active connections Long-lived connections
source Hash of source IP Session persistence
uri Hash of URI Caching proxies
first First available server Minimize active servers

âœ… Best Practice - HAProxy
  • Use HTTPS: Always terminate SSL at HAProxy
  • Enable health checks: Use application-level checks
  • Configure timeouts: Appropriate values for your application
  • Enable stats: Monitor via stats page (protect with auth)
  • Rate limiting: Use stick-tables for DDoS protection
  • Security headers: Add HSTS, X-Frame-Options
ğŸ’¡ Interview Tip

Question: "How would you implement rate limiting in HAProxy?"

Answer: "HAProxy uses stick-tables for rate limiting. You define a stick-table that tracks requests per IP, then create an ACL that checks the request rate. For example: `stick-table type ip size 100k expire 30s store http_req_rate(10s)` creates a table tracking the request rate over 10 seconds. Then `http-request deny deny_status 429 if { sc_http_req_rate(0) gt 100 }` blocks IPs exceeding 100 requests in 10 seconds with a 429 status code. This happens in memory, so it's very fast."
