VLAN Segmentation

Cisco VLAN Configuration with Inter-VLAN Routing 
! =============================================================================
! VLAN SEGMENTATION WITH SECURITY
! =============================================================================

! Create VLANs
vlan 10
 name Users
vlan 20
 name Servers
vlan 30
 name Databases
vlan 40
 name Management
vlan 50
 name Guest

! Configure access ports
interface GigabitEthernet0/1
 description User Workstation
 switchport mode access
 switchport access vlan 10
 switchport port-security maximum 2
 switchport port-security violation restrict
 spanning-tree portfast

interface GigabitEthernet0/10
 description Server Port
 switchport mode access
 switchport access vlan 20

! Configure trunk to router/firewall
interface GigabitEthernet0/24
 description Trunk to Firewall
 switchport mode trunk
 switchport trunk allowed vlan 10,20,30,40
 switchport trunk native vlan 999

! =============================================================================
! ROUTER-ON-A-STICK (Layer 3 routing between VLANs)
! =============================================================================

interface GigabitEthernet0/0
 no shutdown

interface GigabitEthernet0/0.10
 encapsulation dot1Q 10
 ip address 10.10.10.1 255.255.255.0
 description Users VLAN

interface GigabitEthernet0/0.20
 encapsulation dot1Q 20
 ip address 10.10.20.1 255.255.255.0
 description Servers VLAN

interface GigabitEthernet0/0.30
 encapsulation dot1Q 30
 ip address 10.10.30.1 255.255.255.0
 description Databases VLAN

! =============================================================================
! INTER-VLAN ACCESS CONTROL (ACL)
! =============================================================================

! Allow Users to Web servers only
ip access-list extended USERS-TO-SERVERS
 permit tcp 10.10.10.0 0.0.0.255 10.10.20.0 0.0.0.255 eq 80
 permit tcp 10.10.10.0 0.0.0.255 10.10.20.0 0.0.0.255 eq 443
 deny ip 10.10.10.0 0.0.0.255 10.10.20.0 0.0.0.255 log
 permit ip any any

! Block Users from Databases
ip access-list extended USERS-TO-DB
 deny ip 10.10.10.0 0.0.0.255 10.10.30.0 0.0.0.255 log
 permit ip any any

! Allow only Servers to Databases
ip access-list extended SERVERS-TO-DB
 permit tcp 10.10.20.0 0.0.0.255 10.10.30.0 0.0.0.255 eq 3306
 permit tcp 10.10.20.0 0.0.0.255 10.10.30.0 0.0.0.255 eq 5432
 deny ip any 10.10.30.0 0.0.0.255 log
 permit ip any any

! Apply ACLs
interface GigabitEthernet0/0.10
 ip access-group USERS-TO-SERVERS out
 ip access-group USERS-TO-DB out

interface GigabitEthernet0/0.20
 ip access-group SERVERS-TO-DB out

Zero Trust Architecture

Zero Trust is a security model based on the principle "never trust, always verify." It assumes breach and verifies each request as though it originates from an untrusted network.

Zero Trust Components

Identity Provider (IdP)

Centralized authentication with MFA, SSO, and conditional access (Azure AD, Okta, Ping).

Endpoint Detection & Response

Device health verification, compliance checking, threat detection (CrowdStrike, MDE, SentinelOne).

Network Access Control

Microsegmentation, ZTNA, software-defined perimeter (Zscaler, Palo Alto Prisma, Cisco Duo).

Data Protection

Encryption, DLP, classification, rights management (Microsoft Purview, Symantec DLP).

Microsegmentation

Microsegmentation creates granular security policies for individual workloads, limiting lateral movement even within the same network segment.

Config VMware NSX Microsegmentation 
# =============================================================================
# VMWARE NSX MICROSEGMENTATION POLICY
# =============================================================================

# Security Group Definitions
security-group Web-Tier
  membership-criteria:
    - vm-tag equals "tier:web"
    - vm-name contains "web-"

security-group App-Tier
  membership-criteria:
    - vm-tag equals "tier:app"

security-group DB-Tier
  membership-criteria:
    - vm-tag equals "tier:db"

# Distributed Firewall Rules
section: Application-Segmentation
  rule Web-to-App
    source: Web-Tier
    destination: App-Tier
    service: TCP/8080, TCP/8443
    action: Allow
    logging: enabled
    
  rule App-to-DB
    source: App-Tier
    destination: DB-Tier
    service: TCP/5432, TCP/3306
    action: Allow
    logging: enabled
    
  rule DB-Deny-All
    source: Any
    destination: DB-Tier
    service: Any
    action: Deny
    logging: enabled
    applied-to: DB-Tier

# Default Deny (applied last)
section: Default-Deny
  rule Block-All
    source: Any
    destination: Any
    service: Any
    action: Deny
    logging: enabled

Kubernetes Network Policies

YAML Kubernetes NetworkPolicy 
# =============================================================================
# KUBERNETES NETWORK POLICIES (Microsegmentation for K8s)
# =============================================================================

# Default deny all ingress traffic
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-ingress
  namespace: production
spec:
  podSelector: {}
  policyTypes:
  - Ingress

---
# Allow web tier to receive traffic from ingress controller
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-web-ingress
  namespace: production
spec:
  podSelector:
    matchLabels:
      tier: web
  policyTypes:
  - Ingress
  ingress:
  - from:
    - namespaceSelector:
        matchLabels:
          name: ingress-nginx
    ports:
    - protocol: TCP
      port: 8080

---
# Allow app tier to receive from web tier only
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-app-from-web
  namespace: production
spec:
  podSelector:
    matchLabels:
      tier: app
  policyTypes:
  - Ingress
  ingress:
  - from:
    - podSelector:
        matchLabels:
          tier: web
    ports:
    - protocol: TCP
      port: 8080

---
# Allow database tier from app tier only
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-db-from-app
  namespace: production
spec:
  podSelector:
    matchLabels:
      tier: database
  policyTypes:
  - Ingress
  ingress:
  - from:
    - podSelector:
        matchLabels:
          tier: app
    ports:
    - protocol: TCP
      port: 5432

Zero Trust Network Access (ZTNA)

ZTNA provides secure remote access based on identity and context, replacing traditional VPNs with application-specific access.

ZTNA vs Traditional VPN 
┌─────────────────────────────────────────────────────────────────────────────────────┐
│                        ZTNA vs TRADITIONAL VPN                                      │
├─────────────────────────────────────────────────────────────────────────────────────┤
│                                                                                     │
│   TRADITIONAL VPN                                                                   │
│   ═══════════════                                                                   │
│                                                                                     │
│   Remote User ══════► VPN Gateway ══════► Full Network Access                      │
│                            │                                                        │
│                            │   Once connected, user has                            │
│                            │   broad access to internal                            │
│                            │   network resources                                   │
│                            │                                                        │
│                            ▼                                                        │
│              ┌─────────────────────────────┐                                       │
│              │     Internal Network        │                                       │
│              │  ┌────┐ ┌────┐ ┌────┐      │                                       │
│              │  │App1│ │App2│ │App3│ ...  │  User can reach all                   │
│              │  └────┘ └────┘ └────┘      │                                       │
│              └─────────────────────────────┘                                       │
│                                                                                     │
│   Problems:                                                                         │
│   • All-or-nothing access                                                          │
│   • Network-level access (can scan/attack)                                        │
│   • VPN gateway is single point of entry                                          │
│   • Device on network = trusted                                                    │
│                                                                                     │
│                                                                                     │
│   ZTNA (Zero Trust Network Access)                                                 │
│   ═════════════════════════════════                                                 │
│                                                                                     │
│   Remote User ──► ZTNA Broker ──► Identity Check ──► Policy ──► App Access        │
│                        │              │               │                             │
│                        │              │               │                             │
│                        ▼              ▼               ▼                             │
│                   ┌─────────┐   ┌─────────┐    ┌─────────┐                         │
│                   │  User   │   │ Device  │    │ Context │                         │
│                   │  Auth   │   │ Posture │    │  Check  │                         │
│                   └─────────┘   └─────────┘    └─────────┘                         │
│                                                     │                               │
│                                                     ▼                               │
│              ┌─────────────────────────────┐                                       │
│              │     Internal Network        │                                       │
│              │  ┌────┐ ┌────┐ ┌────┐      │                                       │
│              │  │App1│ │App2│ │App3│      │  User can ONLY reach                  │
│              │  │ ✓  │ │ ✗  │ │ ✗  │      │  authorized apps                      │
│              │  └────┘ └────┘ └────┘      │                                       │
│              └─────────────────────────────┘                                       │
│                                                                                     │
│   Benefits:                                                                         │
│   • Application-specific access                                                    │
│   • Identity + device + context verified                                           │
│   • Apps invisible to unauthorized users                                           │
│   • Continuous verification                                                        │
│                                                                                     │
└─────────────────────────────────────────────────────────────────────────────────────┘

Implementation Strategy

Markdown Zero Trust Implementation Roadmap 
# =============================================================================
# ZERO TRUST IMPLEMENTATION ROADMAP
# =============================================================================

## PHASE 1: FOUNDATION (Months 1-3)
─────────────────────────────────────

### Identity
□ Deploy MFA for all users
□ Implement SSO with centralized IdP
□ Create role-based access groups
□ Enable conditional access policies

### Device
□ Deploy MDM/UEM solution
□ Require device enrollment for access
□ Define device compliance policies
□ Deploy EDR to all endpoints

### Visibility
□ Enable logging across all systems
□ Deploy SIEM for centralized logging
□ Create baseline of normal activity
□ Identify critical assets and data


## PHASE 2: NETWORK SEGMENTATION (Months 4-6)
─────────────────────────────────────────────

### Network
□ Map application communication flows
□ Implement VLAN segmentation
□ Deploy next-gen firewall with App-ID
□ Create inter-segment access policies

### Applications
□ Inventory all applications
□ Classify applications by sensitivity
□ Implement application-level auth
□ Enable encryption for data in transit


## PHASE 3: MICROSEGMENTATION (Months 7-9)
──────────────────────────────────────────

### Workload Security
□ Deploy microsegmentation solution
□ Create per-workload policies
□ Implement default-deny policies
□ Enable workload-to-workload encryption

### Data Protection
□ Classify sensitive data
□ Implement DLP policies
□ Enable rights management
□ Encrypt data at rest


## PHASE 4: ZTNA & CONTINUOUS (Months 10-12)
─────────────────────────────────────────────

### Remote Access
□ Deploy ZTNA solution
□ Migrate from VPN to ZTNA
□ Implement per-app access policies
□ Enable continuous authentication

### Continuous Improvement
□ Implement continuous monitoring
□ Automate threat response
□ Regular policy reviews
□ Red team/penetration testing


## SUCCESS METRICS
──────────────────
• Reduction in lateral movement incidents
• MFA adoption rate (target: 100%)
• Device compliance rate (target: 95%+)
• Mean time to detect/respond
• Reduction in attack surface
